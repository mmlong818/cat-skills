# Skillforge 生成器

用一套分阶段方法，把重复工作整理成真正可复用的技能，而不是零散笔记。

## 核心规则

从用户真实、重复出现的需求出发，而不是从目录结构或模板出发。

一个好技能应该让未来的工作变得：

- 更容易被触发
- 更稳定一致
- 更少依赖长篇解释

## 工作流

### 1. 定义这件事

先用一句话说清这个技能：

- 帮什么忙
- 在什么情况下使用
- 用户会说什么来触发它

如果这句话都说不清，技能还不够成熟。

### 2. 收集真实例子

主动询问或推断几种用户真正会提出的请求。

例如：

- “帮我把一个外部 skill 改造成 Codex 版”
- “把重复的发布流程做成一个技能”
- “以后遇到这种文档整理任务，按固定步骤处理”

用这些例子来决定什么应该进入技能，什么不应该。

### 3. 拆分可复用部分

只保留未来确实能反复节省时间的部分：

- `SKILL.md`：工作方法
- `scripts/`：适合脚本化的确定性动作
- `references/`：按需读取的详细资料
- `assets/`：模板、示例、启动资源

不要为了显得完整而硬造文件。

### 4. 强化触发条件

前置描述里的 `description` 是技能触发关键。

一定要明确：

- 技能做什么
- 什么时候该用
- 哪类任务、文件或场景会触发它

不要把触发逻辑藏在正文里。

### 5. 按执行来写

正文应该告诉下一位 Codex 如何成功完成任务：

- 先看什么
- 按什么顺序做
- 要避开什么
- 结束前要验证什么

优先使用短指令、决策规则和实用默认值，而不是理论说明。

### 6. 校验形态

检查这个技能是否：

- 足够短，能快速读完
- 足够明确，能稳定触发
- 足够实用，下一次就能拿来用

凡是不会改变未来执行方式的内容，都考虑删掉。

### 7. 用后再迭代

真实使用后再优化：

- 补缺失步骤
- 把太长的细节挪到 `references/`
- 只有当重复手工劳动出现时再加脚本
- 删除那些看起来正确、实际没帮助的话

## 默认构建顺序

1. 定义任务
2. 收集具体例子
3. 识别可复用资源
4. 创建目录
5. 写精炼的 `SKILL.md`
6. 做校验
7. 在真实使用后继续打磨

更完整的方法清单见 [skillforge-method.md](./references/skillforge-method.md) 与 [zh-cn.md](./references/zh-cn.md)。

## Codex 化改造提示

如果来源是别的系统：

- 保留方法，不保留品牌壳
- 去掉本机没有的工具假设
- 把路径、命令和提示改成 Codex 环境可用
- 优先做一个聚焦技能，不要整套外部系统照搬

## 输出标准

一个完成度合格的技能，至少要留下：

- 清楚的 `SKILL.md`
- 真正需要的资源目录
- 明确可触发的描述
- 一段说明它为什么能让以后更省力的话
