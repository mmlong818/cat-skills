# Requirement Discovery

This is the English companion for `requirement-discovery`.

## Purpose

Turn a vague idea, unclear request, or stakeholder disagreement into a requirement outcome that is clear enough to evaluate, scope, and move forward.

## Best-fit situations

- A request must move forward before it is well defined
- There is an idea, but its value is uncertain
- Business, users, design, and delivery are not aligned
- Value needs to be tested before formal planning

## Core principle

Do not jump to solutions first.

Start by clarifying:

1. who actually has the problem
2. whether the problem is real
3. why it matters now
4. what level of clarity is enough for the next step

## Five stages

- signal capture
- problem clarification
- value judgment
- scope shaping
- execution handoff

## Recommended question order

Ask only one or two sharp questions per round:

1. who is affected
2. in what scenario it happens
3. what goes wrong today
4. what improvement is expected
5. why it matters now

If the discussion stays abstract, force concrete examples:

- When did this happen most recently?
- Who raised the issue first?
- What exactly is lost if nothing changes?
- What outcome would count as effective?

## Output options

- discovery summary
- requirement brief
- value check
- stakeholder alignment note
